# 指摘の抽出ヒューリスティック

ユーザー発言から「修正 / 否定 / 再指示」を抽出する際の判定基準。

## 基本方針

完全な機械的判定は不要。**Claude の言語理解**で「これは指摘か?」を判定する。下記のパターンは精度を上げるためのヒントであり、漏れなく適用するチェックリストではない。

過剰抽出 (偽陽性) は次工程のクラスタリングで「3回以上」フィルタが効くため、最初は**やや広めに拾う**方針でよい。逆に抽出漏れは恒久的な学習機会の喪失になるため避けたい。

## 強いシグナル (ほぼ確実に指摘)

| パターン | 例 |
|---|---|
| 直接否定 | 「違う」「そうじゃない」「それじゃない」 |
| 行動停止要求 | 「やめて」「ストップ」「いったん止めて」 |
| 訂正 | 「○○じゃなくて△△」「○○ではない」 |
| 再指示 | 「もう一度」「やり直して」「最初からやって」 |
| 不満表明 | 「困る」「困った」「うまくいかない」 |
| 期待との乖離 | 「そうじゃなくて」「期待してたのは」「むしろ」 |

## 中程度のシグナル (文脈次第)

| パターン | 例 | 注意点 |
|---|---|---|
| 追加指示 | 「こうしてほしい」「こう書いて」 | 新規タスク指示と区別が必要。直前にアシスタント応答があるなら指摘の可能性が高い |
| 確認・疑問 | 「これで合ってる?」「なんでこうした?」 | 純粋な確認なら指摘ではない。否定的なトーンを伴えば指摘 |
| 部分肯定+修正 | 「だいたい良いんだけど」「そこは違って」 | 後半を抽出対象にする |
| 修飾語 | 「ちょっと」「もう少し」「もっと」 | 単独では弱い。否定形と組み合わさると強い |

## 弱いシグナル (誤検出が多い)

これらは単独では抽出しない。複数の中程度シグナルと組み合わさったときのみ拾う。

- 「うーん」「ふむ」などの躊躇
- 「と思う」「かもしれない」などのヘッジ
- 質問形式の指摘 (「○○じゃない?」)

## 除外すべきパターン (指摘ではない)

| パターン | 例 |
|---|---|
| 新規タスク開始 | 「次は X をやって」 (直前に完了報告がある場合) |
| 純粋な質問 | 「これってどう動く?」 |
| 仕様確認 | 「ここで Y は何を返す?」 |
| 雑談・挨拶 | 「ありがとう」「おつかれ」 |
| 環境報告 | 「コマンドが見つからない」(これは Claude のせいではない) |

## 文脈の取り方

抽出した指摘には次のコンテキストを必ず付ける:

1. **直前のアシスタント発言** (最大 500 文字、超える場合は冒頭と末尾を残して中略)
2. **直前のツール呼び出し** (ツール名と主要引数のみ。例: `Edit /foo/bar.ts`)
3. **直前のターン番号** (jsonl 内の `index` または entry の連番)

これらは後工程の **(a) クラスタリング時に「何の話題か」を判定する材料** と **(b) PR インラインコメントの「推定根本原因」セクションの根拠** に使われる。

## category_hint の付与

Claude が抽出時にゆるいラベルを付ける。クラスタリングのヒントに使うだけなので厳密でなくてよい。例:

- 型安全性
- テスト戦略
- コミット粒度
- 命名規則
- ドキュメント記述
- ツール選択
- 説明の冗長さ
- 確認の頻度
- エラー処理

既存ラベルに当てはまらないなら新規ラベルを作ってよい。後でクラスタリング時に統合できる。

## 偽陽性の典型例 (拾わない)

- **長時間タスクの中の単なる方針変更**: 「やっぱり A 案じゃなくて B 案で」のような、その場の意思決定の変更。これは指摘ではなく仕様変更
- **エラー報告**: 「動かない」「エラーが出てる」だけの発言。Claude の行動への指摘ではなく事実報告
- **環境差異**: 「macOS だとこう動く」のような環境説明。指摘ではなく情報提供

これらを誤って拾うと「ユーザーが頻繁にエラーを報告する」というクラスタができてしまい、改善案として「エラーを出さないようにする」という意味のない PR が生まれる。
