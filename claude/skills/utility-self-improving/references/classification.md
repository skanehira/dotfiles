# 改善対象の振り分け基準

クラスタごとに「どのファイルを更新するか」を決める判定フロー。

## 目次

- [判断フロー](#判断フロー) — 上から順に質問して振り分け先を決める5問
- [ファイル別の振り分け詳細](#ファイル別の振り分け詳細) — `CLAUDE.md` / `rules/` / `skills/` / `agents/` / `hooks/` の守備範囲
- [複数ファイルへの分割](#複数ファイルへの分割) — 1指摘が複数対象に該当するときの原則
- [既存記述との整合](#既存記述との整合) — 重複・矛盾の検出と対処

## 判断フロー

1. **そのパターンは特定の言語/フレームワークに依存するか?**
   - Yes → `claude/rules/backend/<lang>/` or `claude/rules/frontend/<framework>/`
   - No → 次へ

2. **そのパターンは特定のワークフロー (PR作成・レビュー・コミット等) の中で起きるか?**
   - Yes, 既存スキルがある → そのスキルの SKILL.md を更新
   - Yes, 既存スキルが無い → 次へ (新規スキルか rules か判断)
   - No → 次へ

3. **そのパターンは反復作業の手順そのものか? それとも姿勢・思想か?**
   - 手順 → 既存スキル更新 or 新規スキル
   - 姿勢・思想 → `claude/rules/core/` or `claude/CLAUDE.md`

4. **そのパターンは自動化チェックで防げるか? (静的解析・型検証・lint で検知できる類)**
   - Yes → `claude/hooks/` に PostToolUse hook を追加することを検討
   - No → 上のいずれか

5. **判定に迷う**
   - `claude/CLAUDE.md` 追記をデフォルトとする

## ファイル別の振り分け詳細

### `claude/CLAUDE.md`

全タスク共通の姿勢、回答スタイル、調査時の心構えなど。

該当する指摘の例:
- 「説明が冗長すぎる」 → 簡潔さに関する記述を追加
- 「勝手に決めずに確認して」 → 不明点を質問する原則を強化
- 「過程の記録は要らない」 → ドキュメント作成スタイルの記述を追加

**書き方の原則**:
- 1指摘 = 1〜3行の追記が目安
- 既存セクション (「全タスク共通」「実装時」「調査時」など) に追記する
- 新セクションを作るのは、既存セクションに収まらないテーマ (例: コミュニケーション、UI/UX) のときだけ

### `claude/rules/core/`

普遍的な (言語非依存の) 設計・実装・テスト原則。

| ファイル | 守備範囲 | 該当指摘の例 |
|---|---|---|
| `tdd.md` | TDD サイクル、Tidy First | 「テストなしに実装するな」「リファクタと機能追加を混ぜるな」 |
| `design.md` | SOLID, YAGNI, 命名, 凝集度 | 「将来必要かもしれないコードを書くな」「曖昧な名前を使うな」 |
| `testing.md` | テスト命名、振る舞いテスト、アサーション | 「実装の詳細をテストするな」「contains() ではなく完全一致を使え」 |
| `commit.md` | Conventional Commit, コミット粒度 | 「複数の関心事を1コミットに混ぜるな」 |
| `investigation.md` | 網羅的調査、根本原因 | 「表面的な原因で止まるな」 |

### `claude/rules/backend/` `claude/rules/frontend/`

言語/フレームワーク固有のコーディング規約。

該当する指摘の例:
- TypeScript で `any` を避ける → `frontend/typescript/`
- Go で `interface{}` を `any` と書く → `backend/go/`
- Rust で `unwrap()` を避ける → `backend/rust/`
- React の `useEffect` を乱用しない → `frontend/react/`

サブディレクトリが存在しない場合は新規作成する。

### `claude/skills/<既存スキル>/`

特定ワークフローの手順そのものに関する指摘。

該当する指摘の例:
- 「コミットメッセージに body が無い」 → `workflow-commit/SKILL.md` 更新
- 「PR の Test plan が薄い」 → `workflow-create-draft-pr/SKILL.md` 更新
- 「レビュー観点が足りない」 → `workflow-review/SKILL.md` 更新

**注意**: そのスキルが本当に該当するか確認すること。「実装時の指摘」を `workflow-review` に追加してしまうとお互いに無関係な内容が混入する。

### `claude/skills/<新規スキル>/`

既存スキルで吸収できず、かつ反復可能な手順がある場合のみ。**最終手段**。

新規スキル作成の閾値:
- 5 回以上観測されたパターン (3 回ではなく 5 回)
- 既存スキル群に同種のものが無い
- 「次に同じ状況に遭遇したら、このスキルを呼ぶ」と明確に言える

該当する例:
- 「API のレスポンス JSON を curl で確認する手順」 → `utility-api-probe` のような新規スキル候補
- 「特定の OSS ライブラリの更新手順」 → ライブラリ固有スキル

新規スキル作成時は `utility-create-skill` を内部で呼び出す。

### `claude/agents/`

特定の専門判断を切り出した sub-agent。

該当する指摘の例:
- 「設計レビューが甘い」 → architecture-guard agent
- 「型エラーの修正が不正確」 → ts-type-fixer agent (既に fix-lsp-warnings がある)

agent 新規作成は新規スキル同様に保守的に。

### `claude/hooks/`

PostToolUse / Stop / Notification で自動チェックを差し込む。

該当する指摘の例:
- 「テスト書き忘れ」 → 新規ファイル作成時に対応する test ファイル不在を警告する hook
- 「コミット前のフォーマット忘れ」 → PostToolUse で format を強制

hook の言語: 既存の `tdd-guard.ts` に倣い Deno (TypeScript) で書く。

## 複数ファイルへの分割

1 つの指摘が複数の対象に関係することがある。例: 「TypeScript で `any` を避ける」は:
- 言語固有なので `rules/frontend/typescript/types.md`
- 自動チェック可能なので `hooks/lint.ts` でも検出
- 普遍原則として「型安全性を意識する」を `rules/core/design.md` にも書ける

**原則**: **1指摘 = 1ファイル更新**。複数に分けると PR がノイジーになり、レビュー時に判断材料が分散する。一番具体的な場所に書く (上の例なら `rules/frontend/typescript/types.md`)。

## 既存記述との整合

更新前に対象ファイルを Read して、**既に類似の記述がないか**確認する:

- 完全に同じ → 更新しない (skipped ログに「既存ルールと重複」と記録)
- 一部重なる → 既存記述を**差し替え**ではなく**具体化・補強**する
- 矛盾する → 中断してユーザーに判断を仰ぐ (自動で矛盾を解消しない)

矛盾するケースの典型:
- 既存ルール「最小実装を徹底する」 vs 観測パターン「もっと丁寧にエラーハンドリングを書いてほしい」
- これは表面的な矛盾。状況依存 (どこで丁寧にやるか) なので機械的に判定できない。人間に投げる
