## Cloudflare Workers

出典: <https://developers.cloudflare.com/workers/platform/pricing/>

| プラン | 月額 | 込み枠 | 超過 |
|--------|------|--------|------|
| Free | $0 | 10 万リクエスト/日、CPU 10ms/呼び出し | ハード上限 (課金なし・停止) |
| Paid | $5 | 1,000 万リクエスト/月、3,000 万 CPUms/月 | $0.30/100万 req、$0.02/100万 CPUms |

- Free の実質的な制約は「CPU 10ms」(重い処理が不可) と D1 読取枠。**解析・帳票系の機能があるなら初月から Paid 前提**が正しい
- 重い帳票 (Excel 等) の生成は**ブラウザ側で構築する設計**にすれば Workers の CPU 制約自体を回避できる

## Cloudflare D1

出典: <https://developers.cloudflare.com/d1/platform/pricing/> / <https://developers.cloudflare.com/d1/platform/limits/>

| 項目 | Paid の込み枠 | 超過単価 |
|------|--------------|---------|
| 行読取 | 250 億行/月 | $0.001/100万行 |
| 行書込 | 5,000 万行/月 | $1.00/100万行 |
| ストレージ | 5GB | **$0.75/GB/月** |

**ハード上限 (課金では解決できない)**:
- **1 データベース = 10GB**。長期保存 SaaS ではここが「1 DB に収容できるテナント数」を決める
- アカウントあたり 50,000 DB・合計 1TB → テナント別 DB 分割で実質スケール可能
- **単一スレッド処理** (1 クエリずつ)。テナント数が増えて解析クエリが重なると詰まる → DB 分割は容量とスループットを同時に解決する
- Time Travel (バックアップ) は **30 日まで**。法定保存はR2 アーカイブで担保する

Free の枠 (参考): 読取 500 万行/日・書込 10 万行/日・5GB。解析ダッシュボードが蓄積データをスキャンすると読取枠が実質の壁になる。

## Cloudflare R2

出典: <https://developers.cloudflare.com/r2/pricing/>

| 項目 | 値 |
|------|---|
| ストレージ (Standard) | $0.015/GB/月 (**D1 の 1/50**) |
| 無料枠 | 10GB + Class A 100万回 + Class B 1,000万回/月 |
| 取り出し (egress) | **無料** (S3 と違い復旧時のダウンロードも無料) |
| Infrequent Access | $0.01/GB/月 + 取り出し $0.01/GB。**無料枠対象外なので小容量では Standard が得** |

用途: D1 Time Travel (30 日) を超える長期アーカイブ。日次エクスポート (JSONL/SQLite ダンプ) の Cron 1 本で実装できる。

## 認証: AWS Cognito Essentials (東京) — 既定の推奨

出典: <https://aws.amazon.com/cognito/pricing/> / <https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-sign-in-feature-plans.html>

| 項目 | 値 |
|------|---|
| 無料枠 | **10,000 MAU/月** (2024-11 以降の新規プールの Lite/Essentials。そのまま本番プラン) |
| 超過 | **$0.015/MAU** (単価従量型。ティア段差なし) |
| TOTP MFA (認証アプリ) | **全プラン無料** |
| データ所在 | 東京リージョン (ap-northeast-1) 選択可 → 日本国内保存 |
| 注意 | SMS MFA は SNS 別課金 (使わない)。Plus プランは無料枠なし |

### 認証サービス比較の要点 (2026-07 調査)

| サービス | 特徴 | 落とし穴 |
|---------|------|---------|
| Auth0 Essentials | $35 (〜500 MAU) | **ティア段差型**: 1,000 人 $70 → 2,500 人 $175 → 10,000 人 $700。テナント増で跳ねる。Free 25,000 MAU は機能不足 (ログ 1 日・サポートなし) で本番不可。Free の MAU 枠は有料プランに引き継がれない |
| WorkOS AuthKit / Stytch | 100万 / 1万 MAU まで無料 | **米国保存のみ** → 個人情報の越境論点が増える |
| Clerk / Kinde / Supabase | $25/月 定額圏 | Clerk/Kinde は米国/シドニー保存。Supabase は東京可だが Postgres が付いてくる |
| 自前 (Better Auth + D1) | $0、Workers ネイティブ | セキュリティ運用が自己責任。データ所在はアプリ本体と同条件 (悪化しない) |

判断基準: **単価従量型か / 日本リージョンか / TOTP MFA が無料か / 打刻など高頻度操作を認証から切り離せるか** (認証必須の操作をログイン系に限定すると MAU 自体を 1/10 以下にできる)。

## 使用量モデルの既定値 (1 ユーザあたり)

| 項目 | 既定値 | 根拠 |
|------|--------|------|
| API リクエスト | 600 件/人/月 | 記録操作 + 履歴閲覧 + 承認操作 |
| DB 書込 | 1,000 行/人/月 | 業務レコード + 監査ログ + ワークフロー |
| DB 読取 | 50,000 行/人/月 | ダッシュボードのスキャンを人数按分 |
| データ蓄積 | 1.8MB/人/年 (= 150KB/人月) | レコード + 監査ログの増分 |

- あくまで初期仮置き。**実装後に実測して Excel の入力セルを更新する**よう必ず案内する
- 蓄積は人月ベースで計算: `年間人月 = 平均アクティブ × 通常月数 + 繁忙アクティブ × 繁忙月数`

## 他インフラを検討する場合の Web 裏取り手順

1. 候補ごとにサブエージェント (general-purpose + WebSearch/WebFetch) を**並列起動**して調査する
2. 必ず**公式 pricing ページ**を一次ソースにする (第三者ブログの価格は改定に追従していないことが多い)
3. 各候補で確認する 3 点:
   - **無料枠**: 上限値と「本番利用に耐えるか」(機能制限・サポート・ログ保持・ブランディング強制)
   - **課金単位**: 何に課金されるか (MAU / リクエスト / トークン / ストレージ)。**ティア段差型か単価従量型か**
   - **ハード上限**: 課金で解決できない上限 (DB サイズ・リージョン・レート制限)
4. 現在規模と将来規模 (例: MAU 300 / 900 / 3,000 / 10,000) の各時点の月額を表にして比較する
5. データ所在 (日本リージョンの有無) は個人情報を扱う SaaS では必須確認項目
