# utility-hard-task — 難しいタスクを進めるループエンジニアリング

## 中核コンセプト: ループエンジニアリング

難しいタスクを一発で解こうとすると、迷走・検証不足・スタックが起きる。
そうではなく、着手前に **「分解 → 実行 → 検証 → 次判断」のループを設計してから回す**。

このループが機能するのは、各周回で「観測可能な事実」が 1 つ増えるからである。
推測を積み上げるのではなく、検証結果という事実を足場にして次の一歩を決める。
一歩ごとの失敗は小さく、巻き戻しが効き、進捗が常に測定できる。

```
[Step 1] 分解: ゴール定義 → 検証可能な単位に分割 → 順序付け
    ↓
[Step 2] 実行 + 自己検証: 1 単位ずつ実装し、観測ベースで検証
    ↓
[Step 3] 次判断: 検証結果で分岐 (次へ / 修正 / 転換 / エスカレ)
    ↓
    Step 2 に戻る (全単位消化 + ゴール達成でループ脱出)
```

## Step 1: タスク分解

### ゴールと Definition of Done を書き出す

着手前に次の 2 つを 1〜2 文で明文化する:

- **ゴール**: このタスクが完了したとき、何がどう変わっているか
- **DoD (完了の定義)**: 完了をどうやって確認するか (実行するコマンド、観測する出力)

検証手段が特定できないタスクは着手してはならない。「何をもって完了とするか」が
曖昧なままループを回すと、脱出条件のない無限ループになる。不明ならユーザーに確認する。

### 検証可能な単位に分割する

分解の基準は 2 つ:

1. **独立して検証可能**: その単位だけで「できた / できていない」を観測で判定できる
2. **失敗しても巻き戻せる**: その単位が失敗しても、前の単位の成果は無傷で残る

「モデル層を作る」のような成果物ベースではなく、「X を入力すると Y が返ることを
テストで確認できる」のような検証ベースで単位を切る。

### 不確実性の高い順に並べる

未知の API・使ったことのないライブラリ・技術的に成立するか不明な部分を先頭に置く。
理由: 不確実な部分が後半で破綻すると、それまでの作業が無駄になる。
先に小さく試して (spike)、成立を確認してから本実装に入るほうが総コストが低い。

### タスクリストで見える化する

分解した単位を 1 単位 = 1 タスクとして TaskCreate (なければ TodoWrite) で登録し、
1 件ずつ in_progress → completed のサイクルで消化する。頭の中だけで管理すると、
長いセッションで「どこまで終わったか」を見失う。

**分解例** — 「CLI に自動アップデート確認機能を追加」の場合:

```
1. [spike] アップデート情報 API のレスポンス形式を実データで確認
   検証: curl で実 API を叩き、想定フィールドが存在することを観測
2. バージョン比較ロジック (semver 比較が正しい結果を返す)
   検証: 境界ケース含むテストが通る
3. API クライアント (fetch を DI し、fake で応答を差し替え可能)
   検証: fake 応答でパース結果の構造体全体が期待値と一致
4. CLI への組み込み (起動時に確認し、新版があればメッセージ表示)
   検証: 実際に CLI を起動して出力を観測
```

不確実な API 仕様の確認 (spike) が先頭、既存機能への組み込みが最後に来ている。

## Step 2: 実行と自己検証

### 1 単位ずつ実行する

複数の単位を同時に進めない。同時に触ると、検証が失敗したときに
どの変更が原因か切り分けられなくなる。

### 観測ベースで検証する

単位を終えるたびに、次のいずれかで検証する:

- テストを実行して結果を見る
- 実際に動かして出力を観測する (curl / ブラウザ / CLI 実行)
- 期待値を先に宣言してから実行し、結果と突き合わせる

**「コードを書いたから動くはず」は検証ではない。** 観測していないものは
完了と見なさない。期待値は実行前に宣言する — 実行後に結果を見てから
「これで正しい」と後付けするのは検証ではなく追認である。

### デグレを確認する

新しい単位の検証が通っても、変更前に通っていたテスト・動作が
悪化していないかを毎回確認する。修正が別の箇所を壊すのは典型的な失敗パターンで、
最後にまとめて気づくと原因の切り分けに周回全部を費やすことになる。

### 検証不能な項目は明示する

環境やデータの制約で検証できない項目が出たら、隠さずに
「この項目は○○の理由で未検証」と記録し、最終報告に含める。

## Step 3: 次アクション判断

各単位の検証結果を見て、次の 4 分岐で機械的に判断する:

### 検証成功 → 次の単位へ

その単位のタスクを completed にし、次の単位のタスクを in_progress にして Step 2 へ戻る。

### 検証失敗 → 原因を 1 つ特定して修正

失敗したら、まず原因の仮説を 1 つ立て、その 1 箇所だけを直して再検証する。
推測で複数箇所を同時に触ると、何が効いたのか分からなくなり、
偶然通ったコードが残る。

### 同一アプローチ 3 回失敗 → アプローチ転換

同じやり方で 3 回失敗したら、そのアプローチは捨てる。試行錯誤を続けるのではなく:

- 別のアプローチに切り替える
- 公式ドキュメント・実際のコードに立ち返って前提を検証し直す
- ユーザーに詰まっている箇所を具体的に説明して判断を仰ぐ

「あと 1 回やれば通るかもしれない」は 3 回目以降は成立しない。
3 回失敗はアプローチ自体が間違っているシグナルとして扱う。

### 全単位消化 → ゴール判定

全タスクが completed になったら、Step 1 で定義した DoD と突き合わせる:

- **DoD 達成** → ループ脱出。検証結果 (テスト件数、動作確認内容、未検証項目) を添えて完了報告する
- **DoD 未達** → 未達分を新しい単位としてタスク化し、Step 2 へ戻る

タスクを全部消化したことと、ゴールを達成したことは別物である。
最後に必ず DoD に戻って照合する。

## 範囲外 (やらないこと)

- TDD サイクル・テスト作法・コミット規約の詳細: 既存 rules (tdd.md / testing.md / commit.md) に従う。このスキルはループの回し方に集中する
- 単純な 1 ステップ作業: 分解の意味がないタスクにこのループを適用しない
- 要件定義・設計の生成: `/requirements` や `/workflow-spec` の領分
