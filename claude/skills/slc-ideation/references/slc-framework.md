## MVPとの根本的な違い

| 観点     | MVP                          | SLC                          |
| -------- | ---------------------------- | ---------------------------- |
| 視点     | 自己中心（学習のため）       | 顧客中心（使いたいと思える） |
| 完成度   | 不完全さを許容               | 限定範囲内で完全             |
| 失敗時   | 製品自体が不良               | 価値を提供した上での実験     |
| リリース | v0.1（恥ずかしいバージョン） | v1.0（優れたバージョン）     |

## SLCの3要素

### 1. Simple（シンプル）

**原則**: 複雑な製品は迅速に市場投入できない

- スコープを意図的に限定する
- コア価値に特化する
- 機能数を最小限に抑える

**例**: Google Docsは当初Wordの機能の3%程度だったが、「シンプルさと協調編集」というコア価値に特化して成功

### 2. Lovable（愛される）

**原則**: 機能数より感情的な繋がりが重要

感情的繋がりを生む手段：
- 優れたUIデザイン
- 企業文化・透明性
- 深い顧客理解
- ターゲット層との即座の共感

**例**: Herokuは技術者向けコマンドラインを前面に出すことで、ターゲット層（開発者）との即座の繋がりを実現

### 3. Complete（完全）

**原則**: 「恥ずかしいv1.0」ではなく「優れたv1.0」を目指す

- スコープは限定的だが、その範囲内では完全に機能する
- 半端な状態でリリースしない
- 継続開発しなくても価値を保ち続ける完成度

## 成功したSLC製品の例

| 製品               | 初期のシンプルなスコープ     |
| ------------------ | ---------------------------- |
| Snapchat           | タップで撮影、送信後削除のみ |
| WhatsApp           | ステータスメッセージ機能のみ |
| Dropbox            | 単一フォルダの同期機能に特化 |
| Twitter            | 140文字メッセージのみ        |
| Southwest Airlines | 1路線のみで開始              |

## なぜSLCが優れているのか

### 1. 初日から「喜び」と「有用性」を両立

従来の階層（有用性→信頼性→喜び→有意性）を下から積み上げる必要はない。最初から顧客の心を掴みながら段階的に複雑化できる。

### 2. 失敗しても価値がある

- MVPの失敗 = 製品自体が不良
- SLCの失敗 = 価値を提供した上での実験

### 3. 継続性の選択肢

成功時には満足している顧客からの提案を得られる（「修正してほしい」という不満ではなく）。

## 実践的なチェックリスト

### Simpleの確認

- [ ] このスコープで2-4週間以内にリリースできるか？
- [ ] 説明に30秒以上かからないか？
- [ ] コア価値は1つに絞れているか？

### Lovableの確認

- [ ] ターゲットユーザーが「これ欲しい」と即座に感じるか？
- [ ] 感情的な繋がりを生む要素があるか？
- [ ] 競合との差別化ポイントは明確か？

### Completeの確認

- [ ] この範囲内で「完成」と言えるか？
- [ ] 追加開発なしでも価値を提供し続けられるか？
- [ ] ユーザーに「未完成」という印象を与えないか？

## 壁打ち時の問いかけ

1. **スコープ**: 「これをさらにシンプルにできないか？コア価値は何か？」
2. **愛される要素**: 「ユーザーはなぜこれを使いたいと思うのか？感情的な繋がりは何か？」
3. **完成度**: 「この範囲で完全と言えるか？恥ずかしいv1.0になっていないか？」
4. **検証**: 「顧客に聞いてみて、彼らは同意するか？」

## 核心的なメッセージ

> プロダクト開発の中心に**人間性**を置く

顧客を「学習のための道具」として使うのではなく、顧客が「実際に使いたいと思える製品」を作る。
