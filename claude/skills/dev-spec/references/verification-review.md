# 検証手順の確認と補完 (dev-spec フェーズ 9)

## 目的

ゴール (DESIGN.md) と検証手順 (DESIGN_DETAIL_APP.md / DESIGN_DETAIL_INFRA.md) を精査し、実装ループが機械判定できる形に補完する。ゴールが観測不能なまま実装に入ると、完了判定が自己申告になる。

## 情報収集

以下のソースから検証に関する情報を収集する:

1. **DESIGN.md のゴール**: 「ゴール」セクションを読み取り、観測可能性・検証可能性を評価
2. **詳細設計 2 ファイルの検証手順**: DESIGN_DETAIL_APP.md (ローカル / CI 実行系) と DESIGN_DETAIL_INFRA.md (デプロイ・環境依存系) の「検証手順」セクションを読み取り、実行コマンド・操作手順の充実度を評価
3. **既存プロジェクトの検証手段**: テストファイル、CI 設定、Makefile、package.json の scripts を検索
4. **テスト戦略との整合**: DESIGN.md の「テスト戦略 (方針)」と DESIGN_DETAIL_APP.md の「テスト戦略 (具体)」が矛盾していないか

## 充実度の評価

- ゴールが具体的かつ検証可能か (「◯◯が速くなる」ではなく「p95 で 1 秒以内」)
- 手動検証の手順が操作レベルで記述されているか
- 自動検証 (テストコマンド等) が明記されているか
- **検証コマンドが空虚でないか**: `echo` / `true` 等の恒真コマンド、ゴールと無関係なテスト指定は「検証手順あり」に数えない (機械ゲートは存在しか見ないため、ここで内容を確認する。フェーズ 10.5 の設計整合監査でも第三者チェックされる)
- 完了条件チェックリストがゴールを網羅しているか

## 不足の補完

不足がある場合、AskUserQuestion で確認する。具体的な質問例:

- 「この機能の動作確認はどの環境で行いますか? (ローカル / ステージング / 本番)」
- 「パフォーマンス要件の検証方法は? (負荷テストツール / 手動計測 / APM)」
- 「E2E テストは必要ですか? 必要な場合、どのシナリオをカバーしますか?」

## 設計書更新

収集した情報を以下に振り分けて反映する:

- ゴール (観測可能・検証可能な目標) → DESIGN.md の「ゴール」セクション
- 検証手順の具体のうちローカル / CI で実行できるもの (テストコマンド・実機ブラウザ操作) → DESIGN_DETAIL_APP.md の「検証手順」セクション
- 検証手順の具体のうちデプロイ・環境依存のもの (本番 / ステージング疎通・workflow 実行・IaC plan) → DESIGN_DETAIL_INFRA.md の「検証手順」セクション

## 完了チェック (機械判定、承認前に必須)

以下がすべて通るまでこのフェーズを完了扱いにしない。定性的な「充実している」ではなく、コマンドで判定する:

1. **ゴール存在**: `rg -n '^- G[0-9]+:|^G[0-9]+:' docs/DESIGN.md` が 1 件以上ヒットする (0 件のまま承認に進んではならない)
2. **1:1 対応**: 抽出した各 `G<n>` について `rg -n 'G<n> 検証' docs/DESIGN_DETAIL_APP.md docs/DESIGN_DETAIL_INFRA.md` がヒットする (どちらか一方にあればよい)。欠落ゴールは検証手順を補完してから再チェック
3. **G_E2E** (Web / モバイル Web のみ): `G_E2E` のゴールと検証手順 (APP 側) が存在する

この 3 点は下流の dev-impl Step 1 (構造ゲート) でも同じチェックが走り、欠けていると差し戻される。上流であるここで通しておくのが正。

**goals_sha の対象**: ここで確定させるゴール定義行 (`^- G[0-9]+:` 等) と検証手順行 (`G<n> 検証` / `G_E2E 検証`) は、フェーズ 11 の承認スタンプで `goals_sha` (sha256) としてハッシュバインドされる (計算手順は SKILL.md 11.3)。承認後にこれらの行を変更すると dev-impl が `approval_stale` で停止するため、**受入基準の変更は必ずこのフェーズに戻って再承認を経る**。
