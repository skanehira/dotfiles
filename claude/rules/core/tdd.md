# TDD（テスト駆動開発）ルール

すべてのプロダクションコードは Kent Beck の TDD 方法論に従って記述する。

RED (失敗テスト) → GREEN (最小実装) → REFACTOR (グリーン時のみ) のサイクル順序は `hooks/tdd-guard.ts` が tool call レベルで強制する (テスト未着手での実装編集は deny、テスト未実行での停止は block)。本ドキュメントはゲートが機械判定できない「質」の部分を定める。

## 各フェーズの質

- **RED**: 機能の小さな増分を定義する最もシンプルな失敗テストから始める。テスト名は動作を説明する意味のあるものにし、失敗を明確で情報量のあるものにする
- **GREEN**: 現在のテストを通過させる最小限のコードのみ書く。それ以上は書かない
- **REFACTOR**: 重複を排除し、命名と構造で意図を表現し、メソッドを小さく単一責任に保つ。動作する最もシンプルな解決策を使う

## バグ修正プロセス

1. まずバグを示す API レベルの失敗テストを書く
2. 問題を再現する最小のテストを書く
3. 最小限のコード変更で両方のテストを通過させる

## 違反パターン（禁止）

- ❌ 失敗するテストなしにプロダクションコードを書く
- ❌ テストを通過させるために必要以上のコードを書く
- ❌ テストがグリーンでないときにリファクタリングする
- ❌ 「後で追加する」意図でテストをスキップする
- ❌ 同じコミットで構造的変更と動作的変更を混ぜる

**覚えておくこと: テストなしにコードなし。これは交渉の余地がない。**

## Tidy First アプローチ

すべての変更を 2 タイプに分離し、同じコミットで混ぜない：

- **構造的変更 (STRUCTURAL)**: 動作を変更しない再配置 (リネーム・抽出・移動・フォーマット)。変更前後にテストを実行して動作が変わっていないことを検証する
- **動作的変更 (BEHAVIORAL)**: 機能追加・バグ修正・ロジック変更。最初に失敗するテストによって駆動する

両方が必要な場合は常に構造的変更を先に行う。
