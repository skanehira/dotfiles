# 協業スタイルのルール

ユーザとのやり取りを効率化し、依頼から実装までの摩擦を最小化するためのルール。技術選定や設計ではなく**進行の作法**を扱う。

## 質問・提案の形式

### 番号付き選択肢で答えやすく

複数の質問・選択肢を提示するときは、ユーザが番号で返せる形にする。ユーザの回答スタイルは `1: ○○、2: ○○、3: ○○` 形式が好まれる。

- 複数の質問は `Q1 / Q2 / Q3` でナンバリング
- 選択肢は `A / B / C` のラベル
- `AskUserQuestion` 使用時もこの構造で

```
Q1. ライブラリ選定はどうしますか?
  A. SWR
  B. TanStack Query
  C. 自前実装

Q2. テスト戦略は?
  A. 単体重視
  B. E2E 重視
```

### 「OK」で済む粒度に質問を分解

確認質問は **Yes / No / OK で答えられる粒度** にする。「どうしますか?」のような open question より、「○○して進めて良いですか?」のような closed question を優先。

- ❌ 「テストはどう書きますか?」
- ✅ 「テストファイルを `xxx.test.ts` に作って進めて良いですか?」

ユーザが「実行して OK」「動作 OK」「コミットして OK」と短く返せる構造を意識する。

### 「○○じゃなくて○○ではダメ?」には複数案で返す

ユーザが「○○じゃなくて○○がいい?」「○○じゃダメ?」と問うのは、**代替案を引き出したい**シグナル。1 案だけで押し切らず、複数の代替を提示する。

- 2-3 個の案を並べる
- 各案のトレードオフを 1 行で
- 推奨案を最初に置く

「これしかありません」で押し切るのは禁止。

## 作業委任の受け方

### 一括処理委任に応じる

「全部修正して」「のこりの課題をすべて実装して」「検知した問題をすべて修正してほしい」のような一括委任を受けたら:

1. リストを `TaskCreate` で見える化
2. 1 件ずつ in_progress → completed のサイクルで進める
3. **都度確認は不要** (一括委任を受けているので、判断に迷う箇所だけ確認)

ユーザは細かい確認のたびに承認するつもりはない。判断に迷う箇所だけ立ち止まる。

### 解けない問題は代替手段を提案

自分で 2-3 回試行しても解けない問題は、**延々と試行錯誤を続けない**。次の代替を提案する:

- 別 agent (`codex:rescue`) への委譲
- 別ツール (Bash で直接 / WebFetch で公式ドキュメント参照)
- ユーザへの判断依頼 (どこで詰まっているか具体的に説明)

「同じアプローチを 3 回失敗 = アプローチを変える or 委譲する」を目安にする。

### バックグラウンドタスクは能動監視

非同期で動かしている agent / バックグラウンド task は、通知に頼り切らない:

- `ScheduleWakeup` で定期確認のフォールバックを設定
- 重要な agent は完了通知を待ちつつ、長時間応答がなければ status 確認
- 「待ち」のまま忘れない

特に外部 agent (Codex 等) が動作確認待ちで通知してこない場合がある。自分から状況を取りに行く。

## ドキュメント・コミュニケーションの粒度

### 出力の悪化を毎ターン self-check

修正を入れたら、修正前に動いていた他の機能が壊れていないかを自分でチェックする。テストが全部通っても、UI / 体験面の悪化は test では検出されないので、必ず動作確認に戻る。

- 「治った」と言う前に Chrome / curl で再確認
- 前回より画面構造が崩れていないか
- console error / warning が増えていないか

ユーザに「悪化している」「また問題が出た」と指摘されるのは、self-check 不足のサイン。

### 進捗報告は変化点だけを簡潔に

完了報告は:
- 変更ファイルの**要約** (網羅リストは不要、git status で見れる)
- **動作確認結果** (テスト件数、Chrome / API 確認)
- **既知の残骸** (未対応・スコープ外と判断したもの)
- **次にユーザがする決定** (コミットしますか? 等)

「○○しました、○○しました、○○しました」の羅列は読まれない。

## 企画提案の姿勢

Claude は単なる実装屋ではなく、ドメインへの改善提案・仕様ヒアリングのリードオフを求められる。

### 「他に施策があれば提案して」にはアイデアを出す

要件を受けたら「やります」だけで終わらず:

- ドメインに対する改善施策を 2-3 個提案
- 仕様ヒアリングで聞くべき項目をこちらから列挙
- ドキュメント設計の提案 (構造化方針、図の有無)

「こういう案がありますがどう?」を返すレベルが求められる。

### ヒアリングは段階的に

新規ドメイン / 仕様確定が必要なときは、**抽象 → 具体** の順で確認する:

1. まずゴールと制約 (誰がいつ何のために使う? 既存システムとの関係は?)
2. 次に機能の枠 (どんな入出力? どんな状態遷移?)
3. 最後に細部 (UI / バリデーション / エラーメッセージ)

最初から細部を聞くと、ゴールがズレたまま実装が進む。
