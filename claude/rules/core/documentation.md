# ドキュメント作業のルール（設計書・仕様書・README 等）

## 着手前: 種別の明記とフォーマットの検討

- 単一の標準フォーマットを機械適用しない。文書の用途（設計書 / テスト項目書 / 手順書 / README 等）と読者に適した構成を検討・決定してから書き始める
- 文書の冒頭メタ情報に**種別を明記**する（例: `- 種別: 設計書`）。監査時にどの規範を当てるかの判定に使う
- 型別の構造規範が `rules/core/references/doc-formats/<型>.md` にあればそれに従う。無い型は本ファイルの汎用原則のみでよい（規範は実際に書く型から追加式に育てる）
- 議論しながら設計を固める作業では、完成文書を議論中に直接編集しない。決定台帳への追記 → 節目にゼロから再生成の 2 段で進める（進行様式は `/workflow-design-notes`）。議論のたびに完成文書へ書き足すと、全文書き直しルールがコスト的に守れず継ぎ足しに退化する

## 全体整合 > 差分最小

- コードの「外科的変更」ルールはドキュメントには適用しない。相互参照（用語⇄定義、コンポーネント⇄フロー、約束⇄実装物）を持つドキュメントでは、局所差分の最小化より文書全体の整合を優先する
- 概念の追加・削除・置換を含む修正は、該当節だけの局所 Edit で済ませず、全文を読み直して書き直す。字句修正のみ局所 Edit 可
- 概念を削除・置換したときは、旧概念への参照残骸を rg で掃く（存在理由が消えたのに弱い理由を後付けして残すのが典型的な劣化パターン）

## 編集後の整合性パス（毎回実施）

1. **依拠事実の再検証**: 文書が前提とする実環境（設定ファイル・CI・コマンド・ディレクトリ構成）を rg で確認する。過去の調査結果の使い回し禁止（環境は文書と並行して変化する）
2. **固有名の閉包チェック**: 新出のコマンド・ファイル・環境変数・用語は「定義・生成者・消費者」を同時に書く。「後述」「別途定義」のまま宙に浮いた参照を残さない
3. **大きめの改訂後**: `/utility-doc-audit` で fresh context の監査を実行する。書き手自身は「定義したつもり」バイアスで未定義用語を検出できない

## 汎用原則（全型共通）

- 相互参照の多い設計書には、整合性維持の仕掛けとして「用語・成果物一覧」（定義箇所・生成者・消費者の表）と「依拠する外部事実」（確認日付きの前提列挙）の 2 節を入れる
- 表の数値列には単位を、選択肢列には取りうる値の例を書く
- 過程の記録（「以前は〜だったが」）を残さず、最終状態のみを書く
