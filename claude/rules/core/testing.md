# テスト共通ルール

詳細なコード例 → `rules/core/references/testing-examples.md`、言語別テスト命名 → `references/test-naming.md`、テストダブルの使い分け → `references/test-doubles.md`（必要になったら Read する）。

## テスト戦略（上位方針）

- **テストピラミッド**: 単体・結合テストで詳細ロジックを網羅（IO は DI で差し替えて高速実行）。E2E は壊れたら業務に致命傷の golden path だけに絞り、増やさない。単体・結合で代替できる E2E は積極的に削除する
- **シナリオ不足の能動検出**: ユーザに指摘される前に、実装が一段落したら自発的に (1) 実装からユーザシナリオを書き起こし (2) 既存テストのカバレッジマップを作り (3) シナリオ × カバレッジ表で漏れを明示し (4) Critical / Major / Minor で優先度付けして提示する
- **テスト改善は方針が先**: 「何を厚く何を薄くするか」「削除・追加候補リスト」を文書化してから実装に着手する

## 基本原則

- テストは仕様であり実行可能なドキュメント。セーフティネットとして、失敗すべき時にのみ失敗する
- **実装ではなく振る舞いをテストする**。リトマス試験:「このテストが失敗した時、ユーザーにとって何が壊れたか説明できるか」。リファクタリングで壊れるテストは実装詳細をテストしている（設定値のアサーションは NG、ユーザーに見える出力のアサーションは OK）
- テストの独立性: 実行順序に依存しない / 可変状態を共有しない / 作成したリソースを cleanup する / 一意のテストデータを使う

## セルフレビューチェックリスト（テスト実装後に必須）

1. **振る舞いのテストか？** 失敗時にユーザーへの影響を説明できるか
2. **トートロジーではないか？** getter が set した値を返すだけ等は無意味
3. **冗長なヘルパーはないか？** 1 行で済むものを抽出しない
4. **既存パターンを盲目的に踏襲していないか？** コピー元を評価せずに使わない

いずれかに失敗したら、先に進む前にテストを修正する。

## アサーション

- 文字列は**完全一致**（`contains` 等の部分一致は禁止）。メッセージフォーマットの意図しない変更を検出し、期待出力を正確にドキュメント化するため
- 構造体は**全体比較**（個別フィールドのアサーションは禁止）。欠落・余剰・変更されたフィールドを検出するため

## モックと外部依存

可能な限り実際の実装を使う。モックは 外部ネットワーク呼び出し / 分離が必要なファイルシステム操作 / 時間依存の振る舞い / 非決定的な操作 のみ。

## パラメータ化テスト

同じ関数に対しテスト構造が同一（setup → execute → assert）で入力と期待値だけが異なるテストが 2 つ以上できたら、REFACTOR フェーズで変換する。セットアップロジックや検証する振る舞い自体が異なる場合は変換しない。

## テスト構造・命名

- すべてのテストは AAA (Arrange-Act-Assert) パターンに従う
- テスト名は「何を・どういう条件で・どうなるか」の 3 要素を含める（例: `test_parse_version_with_invalid_input_returns_error`）
