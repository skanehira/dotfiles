# ループエンジニアリング

## 定義

**ループエンジニアリング**とは、AI エージェントの「計画 → 実行 → 検証 → 修正」ループが**正しく前進していることを機械的に検証でき、発散したら止められるように、ループの周辺構造を設計する技術**です。

重要な前提が 2 つあります。

1. **ループ本体は既に存在する。** Claude Code や Codex のような現代のコーディングエージェントは、それ自体が「計画 → 実行 → 検証 → 修正」ループの実装です。ユーザーがループを再実装する余地は小さく、エンジニアリングの対象はループの**外側**にあります。
   - 検証器（テスト・lint・型チェック・hooks・CI）
   - 停止・エスカレーション条件
   - 予算（時間・トークン・試行回数）
   - 環境（エージェントが不正できない構造）
2. **プロンプトエンジニアリングの置き換えではない。** ループの各ステップの中身は依然としてプロンプトです。両者は対立ではなく積み重ねの関係にあります。

一言でいえば、**「AI の自走を信じる」から「AI の自走を検証できる」への移行**です。

## 設計要素

ループを設計するとき、次の 7 要素を仕様として書き下します。各要素は「機械判定できる形になっているか」で品質を判断します。

| 要素      | 問い                   | 良い例                                                 | 悪い例                                           |
| --------- | ---------------------- | ------------------------------------------------------ | ------------------------------------------------ |
| Goal      | 完了を機械判定できるか | 「再現テスト + 既存テスト全 green、lint 0 warning」    | 「いい感じに直す」                               |
| Context   | 何を見て判断するか     | Issue 本文、DESIGN.md、失敗テストの出力                | 「コードベース全体」                             |
| Action    | 何を許可・禁止するか   | ソース編集と `npm test` 実行を許可、CI 設定変更は禁止  | 全ツール無制限                                   |
| Feedback  | 結果を誰が評価するか   | `npm test` の exit code（機械）                        | エージェントの「動くはずです」（自己申告）       |
| Memory    | 何を持ち越すか         | 試行ごとの「アプローチ / 結果 / 棄却した仮説」         | 何も記録しない（同じ失敗を繰り返す）             |
| Stop      | いつ止めるか           | 完了条件達成 / 同一テスト 3 回連続失敗 / 試行 5 回超過 | 「終わったら止まる」（無限に「あと少し」が続く） |
| Guardrail | 何をしてはいけないか   | 既存テストの削除・skip を hook で deny                 | 「品質に気をつける」                             |

このうち成否を分けるのは **Feedback（検証器）** と **Stop（停止条件）** です。検証が自己申告だと「それっぽく完了報告するだけの自走」になり、停止条件がないと品質もコストも制御できません。

## 三大失敗モードと対策

自律ループが実務で壊れるパターンは、構造の欠如ではなくほぼ次の 3 つです。

### 1. Reward hacking（検証器の骨抜き）

エージェントがテストを削除・skip・弱体化して green にする。「テストが通った」という信号自体が汚染される、最も深刻な失敗モードです。

**対策: 検証器をエージェントの編集権限の外に置く。** hooks・CI・別プロセスなど、エージェントが書き換えられない層で検証する。例として、この dotfiles の `claude/hooks/tdd-guard.ts` は tool call レベルで RED → GREEN の順序を強制し、「テストを書いたつもり」での実装着手をプロンプトではなくコードで deny します。エージェントへの指示文（「TDD してください」）は破られますが、hook は破れません。

### 2. 発散（同じ失敗の繰り返し）

同一のエラーに対して同じ修正を試み続け、時間とトークンだけが溶ける。

**対策: 試行の記録と打ち切り基準。** 試行ごとに「何を試して何が起きたか」を Memory に残し、「同一アプローチ 2〜3 回失敗で戦略変更、それでも失敗なら人間にエスカレーション」を仕様に書く。

### 3. エスカレーション設計の不在

止まるべきとき（設計の前提が崩れた・不可逆な操作が必要）に止まらず、止まる必要のないとき（些細な選択）に毎回止まる。

**対策: 停止基準を明文化して 3 区分する。** ①設計の破綻 → 停止して人間に判断を仰ぐ、②破壊的・不可逆操作 → 停止、③それ以外の判断 → 選択と根拠をログに残して前進。事後に人間がレビューで乖離に気付ける状態を保つことが、都度確認の代替になります。

## 実用例: Issue 修正ループ

「このIssueを直して」という依頼を、ループ仕様として書き下すとこうなります。

```text
Goal     : Issue #123 の再現テストが green、既存テスト全 green、lint 0 warning
Context  : Issue 本文、関連モジュール、CLAUDE.md
Action   : ソース・テストの編集、テスト実行。既存テストの削除・skip と CI 設定の変更は禁止
Feedback : `npm test` と lint の exit code。自己申告は完了根拠にしない
Memory   : 試行ごとに「アプローチ / 結果 / 棄却した仮説」を記録して持ち越す
Stop     : 完了条件達成 → PR 作成して終了
           同一テストが 3 回連続失敗 → 試行記録を添えて人間にエスカレーション
           試行 5 回超過 → 中断して状況報告
Guardrail: 再現テストより先の実装着手は hook で deny（tdd-guard）
           force push・main への直 push 禁止
```

ポイントは、これが「テストを書く → 実行する → 直す」という**手順の列挙ではない**ことです。手順の列挙は失敗しない前提のパイプラインにすぎません。ループ仕様の価値は、**失敗時の分岐・打ち切り・不正防止**が書かれていることにあります。

## 一言でいうと

**ループエンジニアリングとは、エージェントが嘘をつけず、無限に迷走できず、止まるべきときに止まる構造を設計すること**です。

開発者の実務では「良いループ図を描く力」よりも、**エージェントの編集権限の外にある検証器（テスト・hooks・CI）を堅牢に作る力**と、**停止・エスカレーション条件を仕様として書き切る力**がレバレッジになります。
