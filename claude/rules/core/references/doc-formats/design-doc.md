# 設計書の構造規範

/utility-doc-audit のフォーマット適合チェックの正本。各項目は yes/no + 根拠箇所で判定できる形にしてある。

節構成の正本は dev-spec スキルの `references/design-template.md`。本規範は独自フォーマットを定義せず、「dev-impl の実装ループに入れられる設計書」の要件をテンプレートに重ねて定義する。実装を伴わない検討文書（方針比較・アーキテクチャ調査等）は本型の対象外 — 種別をそれと明記し（例: `- 種別: 設計方針書`）、`rules/core/documentation.md` の汎用原則のみでよい。

## 成果物とパス（違反は major）

1. 概要 = `docs/DESIGN.md`・詳細 = `docs/DESIGN_DETAIL.md` の 2 分冊（dev-impl のパス契約。単一ファイルや `docs/` 外の配置では dev-impl を起動できない）
2. 節構成は design-template.md に従う。プロジェクトの性質上該当しない節（非機能要件・データ設計等）は「N/A: 理由 1 文」を明記して省略できる。ゴール・検証手順・アーキテクチャ設計の 3 節は N/A 不可

## dev-impl ゲート（機械判定・違反は major）

| 要件 | 判定 |
| --- | --- |
| ゴール定義 | `rg -n '^- G[0-9]+:\|^G[0-9]+:' docs/DESIGN.md` が 1 件以上 |
| ゴール ↔ 検証の 1:1 | 抽出した各 `G<n>` について `rg -n 'G<n> 検証' docs/DESIGN_DETAIL.md` がヒット |
| PoC 未解決なし | `rg -n '<!-- POC_NEEDED: .* -->' docs/DESIGN.md docs/DESIGN_DETAIL.md` が 0 件 |
| G_E2E | Web / モバイル Web なら `G_E2E` のゴールと検証手順がある（CLI / API のみのプロダクトは省略可） |

- ゴールは 1 件 1 行 `- G<n>: ...`。ユーザー視点で観測可能・検証可能な記述（「速くなる」ではなく「p95 で 1 秒以内」）
- 検証手順は自動系 `G<n> 検証: <bash コマンド>` / 手動系 `G<n> 検証 (手動): <操作手順>` 形式（DESIGN_DETAIL.md の「検証手順」節）

## 整合性の仕掛け（違反は major/minor）

design-template.md に無い追加必須節。dev-impl は特定パターンの rg しか行わないため、節の追加はゲートを壊さない:

1. **決定一覧**（DESIGN.md・結論先行の表）: 「決定 / 理由 1 文 / 詳細の参照先」を 1 決定 = 1 行で列挙し、読者が本文を読まずに決定の全体を把握できる。同じ決定の説明を複数節に重複させない（詳細節からはこの表を参照する）
2. **用語・成果物一覧**（DESIGN_DETAIL.md）: 新出の固有名（コマンド・ファイル・環境変数・ラベル等）の「定義箇所・生成者・消費者」の表
3. **依拠する外部事実**（DESIGN_DETAIL.md・確認日付き）: 文書が前提とする実環境の列挙。改訂時はこの節を再検証する

## 構造（違反は major/minor）

1. 冒頭メタ情報: タイトル直後に 作成日・ステータス・種別・対象 があるか
2. 抽象→具象の順序: 目的/背景 → ゴール → 全体像（図）→ 構成要素 → 詳細仕様 → 導入・テスト・リスク の順で、逆流（詳細の後に全体像）がないか
3. 全体図: 構成要素が 3 つ以上あるなら全体アーキテクチャ図があるか
4. 表の完全性: 数値列に単位、選択肢列に取りうる値の例があるか（「詳細は別途確認」単独の記述がないか）
5. リスク節: 未解決事項が「リスク / 対策・状態」のセットで列挙されているか

## 可読性（違反は suggestion）

- 1 節 1 主題（節の要約が「〜と〜と〜」にならない）
- 各節は結論の一文で始まり、根拠・詳細が続く
- 図・表を伴わない 5 段落超の連続散文がない
- 過程の記録（「以前は〜だったが」）を含まない（最終状態のみ）
